## 二、架构差异对照

| 维度 | 模式一（三省六部制） | 模式二（装修队） |
|------|---------------------|-----------------|
| **通信方式** | SharedBus 发布/订阅 | TeamMessageCenter 队列（队长↔队员） |
| **决策者** | IntelligentScheduler（三层：历史→LLM→规则） | TeamLeader（LLM 唯一决策者） |
| **Agent 关系** | 同事（自治，可互相通信） | 层级（队长→队员，队员间不可通信） |
| **协作策略** | 5种（流水线/主从/评审/拍卖/混合） | 1种（队长拆任务 + 队员执行） |
| **跨次学习** | KEPA 闭环（支持） | 不支持 |
| **容错机制** | CircuitBreaker + 快照恢复 | LLM 不可用回退单兵模式 |
| **任务分解** | 调度器自动按复杂度选模式 | LLM 分析后决定团队组成 |
| **代码位置** | `core/multi_agent_v2/` | `core/agents/group_collaboration.py` |
| **核心文件** | ~96K（intelligent_scheduler.py） | ~925 行（group_collaboration.py） |

## 三、关键节点追踪（32 节点摘要）

### 模式一链路
```
任务到达
  → _analyze_task()        [复杂度/关键词/依赖分析]
  → _select_collaboration_mode()  [三层决策: 历史→LLM→规则]
  → _match_and_assign()     [CapabilityMatcher 能力评分]
  → _create_execution_plan()  [执行顺序编排]
  → SharedBus 发布调度结果
  → Agent 自治执行 (think → act → memory)
```

### 模式二链路
```
任务到达
  → TeamLeader.run()
    → _plan_team()          [LLM 分析任务, 决定团队组成]
    → _assemble_team()      [动态创建 1~5 队员]
    → _dependency_layers()  [依赖拓扑排序]
    → _lead_team()          [按依赖分层执行]
      → 并行: 同层队员
      → 串行: 跨层依赖
    → _listen_loop()        [实时监听队员消息]
    → _collect_results()    [聚合 + LLM 总结]
```

## 四、发现的注意点

1. **调度器模式选择偏差（测试 3）**: IntelligentScheduler 的兜底规则只看复杂度数值，不看 keyword 语义，导致部分任务的协作模式与预期不一致。例如 pipeline 任务（复杂度 0.4）走成 hybrid。
2. **模式二 LLM 依赖**: TeamLeader 的组队、拆任务、总结重度依赖 LLM。LLM 不可用时回退到单兵模式（1 个 general 队员）。
3. **记忆机制差异**: 模式一的 Agent 有独立的 Mind/Memory 系统，模式二的队员通过 TeamMessageCenter 向队长汇报，自身不保留长期上下文。
4. **可测试性**: 模式二的可独立测试部分（消息中心、分层算法、结果聚合）表现正常，核心逻辑与 LLM 解耦良好。

---

## 五、结论

两种模式各有所长：
- **模式一**适合复杂、长生命周期、需要跨次学习的场景（大批量标准化任务）
- **模式二**适合灵活策略的单一复杂任务，实现简单但缺乏持久化学习能力
- 当前测试在无 LLM 环境下全部通过，核心逻辑骨架完整
