1 V1 角色分工型 — 问题诊断
  • 🔴
    致命 调度器单点瓶颈 + 单点故障

    AgentScheduler 是唯一的入口和分发点，所有任务经过它。它挂了整个系统瘫痪。无冗余、无备份。

    文件: core/agent_architecture.md

  • 🔴
    致命 Mock 实现无实际价值

    agent_system.py 中 StrategistAgent/ExecutorAgent/ResearcherAgent 全是模拟实现，用字符串匹配代替实际 LLM 推理。Reflector 用关键词匹配推测失败原因。不能用于生产。

    文件: core/agent_system.py:106-333

  • 🟠
    严重 无 Agent 间通信

    调度器单向分发任务给 Agent，Agent 之间不能互相通信。信息孤岛，无法协作解决复杂问题。

  • 🟠
    严重 固定角色不可扩展

    5 个 Agent 角色固定（检查/爬虫/漏洞/总结/前端），需要新增角色时必须修改调度器核心代码。

  • 🟡
    一般 死代码/空壳模块

    agent_coordinator.py、multi_agent_system.py 都是向后兼容空壳。核心逻辑委托给 V2，但入口不清。

    文件: core/agent_coordinator.py, core/multi_agent_system.py

  • 🟡
    一般 无失败处理机制

    Agent 执行失败后无自动重试、无降级、无熔断。"单 Agent 故障不影响其他" 是 feature 也是 bug — 故障 Agent 的任务直接丢失。

2 V2 智能协作型 — 问题诊断
  • 🔴
    致命 单体文件过大：职责未分离

    intelligent_scheduler.py (96KB) 集成了任务理解、复杂度评估、资源评估、Agent 匹配、模式选择、流程编排、结果聚合。任何一处修改风险波及全模块。base_agent.py (79KB) 同理。

    建议按职责拆分为 4-5 个独立的调度子模块

  • 🔴
    致命 协作策略双路执行混乱

    TaskExecutor.execute() 遍历 assigned_agents 直接调用 agent.act()，但 PipelineStrategy/MasterSlaveStrategy 也有自己的 execute() 和 _execute_agent_task()。调度器分配的协作模式在 TaskExecutor 中被忽略 — 两条路径互不感知。

    文件: infrastructure/task_executor.py:55-80 vs orchestration/collaboration/strategies.py

  • 🟠
    严重 AgentPool 形同虚设

    SimpleAgentPool 就是一个包装的 dict。无健康检查、无负载感知、无自动伸缩、无心跳检测。acquire() 返回第一个可用 Agent，不做任何择优。

    文件: infrastructure/agent_pool.py，仅 59 行

  • 🟠
    严重 LLM 降级策略脆弱

    Mind.think() 3 次重试后降级到 _think_simulated()，但模拟思考返回的 plan 和 reasoning 全是占位字符串，无法用于实际执行。LLM 挂了系统等于废了。

    文件: agents/base/base_agent.py:237-264

  • 🟠
    严重 结果聚合是浅拼接

    _aggregate_slave_results() 把所有结果 list.append 到一起返回。无冲突消解、无去重、无质量评分、无跨 Agent 一致性校验。

    文件: strategies.py:348-360

  • 🟡
    一般 调度模式选择偏差

    测试报告发现：调度器只看复杂度数值，忽略 task keyword 语义，导致 pipeline 任务（复杂度 0.4）被误配为 hybrid 模式。

    文件: 两模式测试报告.md:75-76

  • 🟡
    一般 无跨会话记忆持久化

    Agent 的短期/长期/情景记忆全部在内存中，重启后丢失。KEPA 闭环没有真实的"学习→沉淀"存储后端。

  • 🟡
    一般 MCP 工具缓存重建

    _build_mcp_tool_cache() 每次启动时扫描全部 mcp 目录，不持久化，不增量更新。大项目启动慢。

  • 🟢
    轻微 大量 JSON 序列化/反序列化

    Agent 间通信、任务结果传递频繁 json.dumps()/loads()。高并发下是 CPU 浪费。缺少 typed message 传递。

  • 🟢
    轻微 日志噪音大

    大量 logger.info + 手写 [DEBUG] 日志混在一起。TaskExecutor 有 7 行 [DEBUG] 前缀的临时调试日志未清理。

    文件: infrastructure/task_executor.py:56-73

3 V1 优化方案 — 轻量级升级路径

V1 的核心问题在架构层面（单体调度器、无通信、Mock），修补意义有限。建议做"最小可行升级"后迁移到 V2：

优化 1：替换 mock 为真实 LLM 调用 ✦ P0

将 agent_system.py 中的 decompose_task()、execute_task() 等模拟方法改为调用 LLM。复用 core/engine/llm_backend.py 中的 LLMRouter。去除关键词模式匹配和占位 JSON 响应。

文件: agent_system.py:106-134, 147-176

优化 2：引入 Agent 间消息总线 ✦ P1

V1 有 agent_communication.py 定义的 AgentCommunicationCenter（主题订阅/发布/请求-响应），但 V1 的 Agent 从未使用它。将 V1 Agent 接入该通信中心，实现去调度器的点对点协作。

文件: agent_communication.py (已有基础设施)

优化 3：移除空壳模块 + 统一入口 ✦ P2

agent_coordinator.py、multi_agent_system.py 都是空壳委托到 V2。清理死代码，在 __init__.py 层做统一路由：V1 入口→直接报 deprecated 提示迁移到 V2。

4 V2 优化方案 — 核心优化

优化 1：调度器拆分为微服务层 ✦ P0

将 intelligent_scheduler.py (96KB) 拆分为独立模块：

新模块职责原代码范围
task_analyzer.py任务意图理解、复杂度/关键词/依赖分析~200行
mode_selector.py协作模式选择（历史统计→LLM→规则 三层决策）~150行
capability_matcher.pyAgent 能力评分、负载感知、最优分配~200行
execution_planner.py执行顺序编排、依赖拓扑排序~100行
result_aggregator.py结果合并、冲突消解、去重、质量评分~100行

改造后调度器变为 协调器，只负责任务编排和子模块路由，不再是大而全的单体。

优化 2：统一执行路径 — 消除双路混乱 ✦ P0

当前有两套执行逻辑：

  • TaskExecutor.execute() — 直接调 agent.act()，忽略协作模式
  • PipelineStrategy/MasterSlaveStrategy.execute() — 各自实现执行逻辑

优化方案：TaskExecutor 不再直接执行，而是 委托给选中的 CollaborationStrategy。TaskExecutor 只负责超时控制、状态更新、异常兜底。

TaskExecutor.execute() → schedule_result.strategy.execute() → strategies.execute_with_agents()

优化 3：AgentPool 升级为生产级 ✦ P1

替换 SimpleAgentPool（59行）为 ProductionAgentPool：

能力现状目标
健康检查无每30秒心跳检测，连续3次失败标记为 unhealthy
负载感知无追踪每个 Agent 当前执行任务数 + 队列长度
自动伸缩无队列积压 > 阈值时自动创建新 Agent 实例
择优分配返回第一个综合评分：负载×0.4 + 成功率×0.3 + 匹配度×0.3
优雅下线直接 remove等待当前任务完成后再移除

优化 4：结果聚合智能升级 ✦ P1

当前 _aggregate_slave_results() 做的是 list.append，需要升级为：

  • 去重：对多 Agent 返回的相似结果做相似度去重（编辑距离/embedding）
  • 冲突消解：Agent 结论不一致时，按置信度/历史成功率投票
  • 质量评分：每个结果附带 quality_score（0-1），低于阈值的丢弃
  • 总结摘要：对聚合后的结果调用 LLM 生成简洁摘要（而非返回原始 JSON 数组）

优化 5：跨会话记忆持久化 ✦ P2

当前 Agent 记忆（短期/长期/情景）全部在内存中。引入持久化层：

  • 短期记忆：内存 in-memory（当前实现，无需改）
  • 长期记忆：SQLite 持久化，按 agent_id + task_type 索引
  • 情景记忆：嵌入向量存储，用于相似任务检索
  • KEPA 闭环：Agent 反思提取的经验教训写入 long-term memory，下次同类任务自动注入 prompt

可复用已有 infrastructure: core/memory/ 和 core/search/rag_search_engine.py

优化 6：可观测性加入 ✦ P2

清理 logger.info + 手写 [DEBUG] 的混乱局面：

  • 引入结构化日志：每条日志带 trace_id、agent_id、task_id、duration_ms
  • 关键链路加 timing decorator：自动记录每个阶段的耗时
  • 移除所有以 [DEBUG] 为前缀的手写日志（task_executor.py:56-73）
  • MonitorAgent 接入 Prometheus 指标 + 实时 Dashboard

优化 7：MCP 工具缓存预热 ✦ P2

将 _build_mcp_tool_cache() 改为：启动时异步预热 + 文件缓存 + 文件监听增量更新。避免每次请求重建。

优化 8：调度模式加入语义分析 ✦ P1

在 mode_selector.py 的规则兜底层加入 keyword 语义分析。例如检测到 pipeline 关键词（"排序""依次""按步骤"）即使复杂度不高也走流水线模式。修复测试报告发现的问题。
