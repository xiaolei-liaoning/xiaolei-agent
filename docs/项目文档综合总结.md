## 核心项目: 小雷版小龙虾AI Agent

### 项目简介

**小雷版小龙虾 AI Agent** (xiaolei-agent) 是一个工业级AI智能助手系统,提供类似Claude Code的交互体验。系统采用多Agent协作架构,支持技能扩展、工作流编排和智能降级机制。

**版本**: 3.3.1  
**启动命令**: `python main.py` (端口8001)  
**CLI入口**: `python cli.py`

### 核心功能

#### 1. 多Agent系统

系统包含6种专业化Agent:

| Agent类型 | 职责 | 特点 |
|-----------|------|------|
| **MasterAgent** | 任务调度与协调中心 | 任务分解、结果聚合、流程控制 |
| **WorkerAgent** | 执行具体任务 | 专业化方向(爬虫/分析/处理) |
| **ExpertAgent** | 领域专家 | 安全/性能/通用知识,高专业度 |
| **ReviewerAgent** | 质量审查 | 准确性/完整性/一致性评审 |
| **PlanningAgent** | 智能规划 | 任务拆解与路径规划 |
| **CharacterAgent** | 人物角色 | 李白/Linus等角色扮演 |

#### 2. 智能降级机制

```
工具优先策略 → 代码生成降级 → 安全沙盒执行
     ↓                ↓              ↓
预配置工具      LLM自动生成    隔离环境运行
不可用时触发   Python/Shell脚本 资源限制保护
```

**关键特性**:
- 超时控制、内存限制、模块白名单
- 当无现成工具时,LLM自动生成解决方案代码
- 所有生成代码在隔离沙盒中执行,防止安全风险

#### 3. CLI命令系统

**基础命令**:
- `/help` - 显示帮助
- `/run` - 执行智能工作流
- `/chat` - 进入聊天模式
- `/think` - 切换思考模式
- `/status` - 系统状态
- `/history` - 历史记录

**工具命令**:
- `/mcp` - MCP服务器管理
- `/agent` - Agent管理
- `/scrape` - 数据爬取(微博/知乎/B站/GitHub)
- `/analyze` - 数据分析
- `/game` - 小游戏(guess/rps/dice)
- `/fun` - 趣味工具(joke/fact/fortune)
- `/review` - 代码审查
- `/smart` - 智能多Agent协作

**MCP服务器命令**:
- `/mcp list` - 列出已连接服务器
- `/mcp connect <server>` - 连接指定服务器
- `/mcp call <server> <tool>` - 调用工具
- `/mcp quick <tool>` - 快速调用当前服务器工具

支持的MCP服务器包括:
- **本地服务**: calculator, weather, fun, joke, text-processing, file-ops
- **NPM包**: playwright, github, brave-search, slack, discord, chroma, sqlite等

#### 4. 技能系统

**内置技能**:

## 技术架构

### 小雷版小龙虾Agent架构

#### 两种执行模式对比

**模式1: multi_agent_v2 (固定角色多Agent系统)**

```
IntelligentScheduler (协作策略: pipeline/masterslave/review/auction/hybrid)
         │
         ▼
GlobalContextCenter (全局共享上下文)
         │
    ┌────┼────┐
    ▼    ▼    ▼
Master Worker Reviewer
(Mind + Memory + LLM Router)
         │
         ▼
CommunicationCenter (消息总线)
```

**数据流**:
```
用户请求 → MasterAgent.think() (LLM理解)
         → MasterAgent.decompose() (拆分子任务)
         → CommunicationCenter分配
         → WorkerAgent.execute() (调用ToolRegistry/SkillRegistry)
         → WorkerAgent.reflect() (LLM反思)
         → ReviewerAgent.review() (质量评审)
         → MasterAgent.aggregate() (汇总输出)
```

**Agent间通信**:
- publish/subscribe模式
- send_direct点对点通信
- request/response双向通信

**模式2: agency_agent.py (动态Agent系统)**

**单Agent模式**:
```
AGENCY.md (人格/规则) + ToolRegistry + SkillRegistry
         │
         ▼
System Prompt (可用工具列表 + 调用规则)
         │
         ▼
LLM Router
    ├─ 直接回答 → 返回文本
    └─ 调用工具 → 执行 → 可选反思 → 结果回LLM → 继续循环
```

**团队模式 (run_team)**:
```
队长(LLM)分解任务 → SharedContext (消息总线)
                              │
                    ┌─────────┼─────────┐
                    ▼         ▼         ▼
                  队员1     队员2     队员3
                  (并行执行,带反问+反思)
```

频道分类:
- 指挥频道: 队长→开始/指令
- 成果频道: 队员→完成/结果
- 告警频道: 异常通知

#### 核心模块结构

```
core/
├── agents/              # Multi-agent v2系统 (Master/Worker/Expert/Reviewer)
├── engine/              # LLM后端/推理引擎/技能分发器
├── handlers/            # 请求处理器 (single_step/multi_step/code_fallback)
├── infrastructure/      # 数据库/缓存/配置/DI容器
├── memory/              # 短期记忆/角色记忆/向量记忆
├── monitoring/          # 系统指标/告警
├── multi_agent_v2/      # 新多Agent编排层
├── workflow/            # 自动化和XML工作流处理
└── skills/              # 技能处理器 (web_scraper/data_analysis等)

cli/
├── base.py              # CLI基础设施
├── command_parser.py    # 命令解析
├── agent_tools.py       # Agent交互工具
└── logging_system.py    # 彩色时间戳日志

api/                     # FastAPI路由 (v1/routes/schedule/workflow)
tests/                   # 测试套件
```

#### 核心处理流程

```
1. 用户请求 → CLI command_parser.py 解析输入
2. Handler选择 → single_step_handler.py 或 multi_step_handler.py
3. 任务处理 → core/tasks/ 任务处理器/规划器
4. 工具执行 → core/engine/skill_dispatcher.py 分发到技能
5. 代码生成降级 → 工具未找到时,LLM在sandbox_executor.py生成代码
6. 多Agent编排 → core/multi_agent_v2/ 协调Agents
```

## 问题与优化

### 过度设计问题分析

#### 大文件问题

| 文件 | 大小/行数 | 问题 |
|------|-----------|------|
| `core/handlers.py` | 1149行 | 单个文件过大,职责过多 |
| `core/engine/skill_dispatcher.py` | 45KB / 1042行 | 关键词硬编码,维护困难 |
| `core/engine/reasoning_engine.py` | 82KB | 逻辑过于复杂,抽象层次过深 |

#### 核心问题

**1. handlers.py - 职责过多**
包含功能:
- 全局状态管理
- BFS上下文记忆
- 工作流处理
- 多步/单步任务处理
- 闲聊处理
- 数据持久化
- 代码生成降级机制(150行!)

**2. skill_dispatcher.py - 硬编码问题**
- 100+行硬编码关键词列表
- 10+个技能配置全部在代码中
- 维护和扩展性差

**3. reasoning_engine.py - 过度设计**
- 5阶段Deep Thinking Protocol框架
- 复杂度矩阵(ThinkingDepth/TaskComplexity/ImpactLevel)
- 质量标准检查器(5个标准,每个有评分)
- LRU缓存机制 + 过期清理
- 7个思考步骤显示逻辑
- 迭代优化逻辑(最多3次迭代)

### 重构方案

#### 阶段1: 高优先级(立即执行)

**1.1 拆分 handlers.py**

目标: 拆分成小的、职责单一的模块

```
core/
├── handlers/
│   ├── __init__.py              # 兼容层
│   ├── global_state.py          # 全局状态管理
│   ├── workflow_handler.py      # 工作流处理

```

目标: 分离配置和逻辑
- 保留核心匹配逻辑(match_skill/is_multi_step/extract_params)
- 将硬编码关键词移到配置文件
- 提供默认配置,保持向后兼容

#### 阶段2: 中优先级(可选优化)

**2.1 简化 reasoning_engine.py**

目标: 保留核心功能,简化过度设计部分

可选简化:
- 移除复杂度矩阵
- 简化质量标准检查器

## 使用指南

### 快速开始

#### 1. 提交任务

```python
import asyncio
from core.multi_agent_system import agent_scheduler

async def main():
    # 启动所有Agent
    await agent_scheduler.start()
    
    try:
        # 提交检查任务
        check_result = await agent_scheduler.submit_task("check", {"url": "https://example.com"})
        
        # 提交爬虫任务
        scrape_result = await agent_scheduler.submit_task("scrape", {"url": "https://example.com"})
        
        # 提交人物SKILL任务
        character_result = await agent_scheduler.submit_task("bestfriend", {"message": "你好,闺蜜"})
        
        # 等待任务完成
        await asyncio.sleep(2)
        
        # 获取任务状态
        check_status = await agent_scheduler.get_task_status(check_result["task_id"], check_result["agent_type"])
        print(f"检查任务状态: {check_status.status.value}")
        print(f"检查任务结果: {check_status.result}")
        
    finally:
        # 停止所有Agent
        await agent_scheduler.stop()

if __name__ == "__main__":
    asyncio.run(main())
```
