| 无幻觉编造 | 10分 | 专门检测虚构内容 |

**代码示例**:
```python

评分维度及权重：
1. 事实准确无错误（40分）
2. 逻辑完整通顺（30分）
3. 贴合用户问题（20分）
4. 无幻觉编造（10分）
```

### 2. 循环迭代优化 ✅

**实现位置**: `core/self_check_middleware.py` - `check_and_optimize()`

**核心逻辑**:
```python
while retry_count <= self.max_retry:
    # 1. 生成回答
    current_answer = await generate_func(user_query, context)
    
    # 2. 评审打分
    check_result = await self._evaluate(current_answer, user_query)
    
    # 3. 判断是否达标
    if check_result.score >= self.pass_score:
        break
    
    # 4. 基于反馈重新生成
    retry_count += 1
```

**特性**:
- ✅ 防死循环机制（max_retry限制）
- ✅ 智能反馈传递（问题和优化建议）
- ✅ 完整历史记录（每轮得分、问题、建议）

### 3. 异步兼容设计 ✅

**实现位置**: 所有核心方法均为async

```python
async def check_and_optimize(
    self,
    user_query: str,
    generate_func: Callable[[str], Any],
    context: Optional[Dict[str, Any]] = None
) -> SelfCheckResponse:
    """完全异步的自检流程"""
```

**优势**:
- ✅ 不阻塞主线程
- ✅ 适配现有FastAPI架构
- ✅ 支持高并发场景

## ⚠️ 注意事项

### 最佳实践

1. **合理设置合格线**
   - 一般场景：80分
   - 重要任务：85-90分
   - 关键业务：90-95分

2. **控制重试次数**
   - 推荐：2-3次
   - 最多：不超过5次

3. **监控统计信息**
   - 定期检查通过率
   - 调整配置优化性能

4. **异常降级处理**
   - 始终提供fallback方案
   - 记录失败日志

### 已知限制

1. **性能开销**
   - 每次自检需要额外1-2次LLM调用
   - 耗时增加100-200%

2. **依赖LLM稳定性**
   - 评审质量取决于LLM能力
   - 需要LLM能理解评分标准

3. **格式解析风险**
   - 依赖LLM严格遵循输出格式
   - 已实现容错机制
