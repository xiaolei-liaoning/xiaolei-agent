## 1. **真正的多Agent架构** - 每个Agent都是"独立的人"

**以前的问题**：以前的系统像一个机器人干所有活，只是换了不同的工具。

**现在的实现**：
- 每个Agent就像一个独立的员工，有自己的：
  - **身份证**（agent_id）：唯一标识
  - **大脑**（Mind）：有自己的目标、信念、意图
  - **记忆**（Memory）：短期记忆记最近的事，长期记忆存经验，工作记忆处理当前任务
  - **技能清单**（Capabilities）：声明自己能干什么
  - **状态机**：知道自己是空闲、忙碌还是停止

**支持的Agent类型**：
| Agent类型 | 职责 |
|-----------|------|
| CHECKER | 结果校验 |
| SCRAPER | 网页抓取 |
| VULNERABILITY | 安全漏洞检测 |
| SUMMARIZER | 文本总结 |
| DATA_ANALYSIS | 数据分析 |
| NLP | 自然语言处理 |
| TEXT_ANALYZER | 文本拆解提取总结 |
| PLANNING | 任务规划 |

**举个例子**：
MasterAgent接到任务后，不是直接调用函数，而是：
1. 先思考："这个任务复杂吗？"
2. 回忆："以前遇到过类似任务，当时是怎么分解的？"
3. 决策："我需要拆成3个子任务"
4. 通知其他Agent："谁来帮我干这些活？"

每个Agent都有自己的"性格"和"专长"，不是简单的工具调用。

## 2. **深度思考引擎** - 会深度分析的AI

**核心思想**：执行复杂任务时，像人类一样进行多阶段深度思考。

**5阶段思考框架**：
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  1.问题理解  │ → │  2.信息收集  │ → │  3.方案设计  │
└─────────────┘    └─────────────┘    └─────────────┘
       │                 │                 │
       ▼                 ▼                 ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  4.执行验证  │ → │  5.反思优化  │    │  质量标准检查  │
└─────────────┘    └─────────────┘    └─────────────┘
```

**思考深度级别**：
- Level 1: Quick（快速模式，< 5秒）- 简单问题
- Level 2: Standard（标准模式，5-30秒）- 中等问题  
- Level 3: Deep（深度模式，> 30秒）- 复杂问题

## 3. **智能反思机制** - 会自我检讨的AI

**核心思想**：执行过程中不断问自己"我做得对吗？需要调整吗？"

**什么时候反思**：
- ❌ 失败了 → 立即反思
- ⏰ 超时了（超过预期2倍时间）→ 反思
- 😕 置信度太低（<60%）→ 反思  
- 📅 每做3步 → 例行检查

**反思流程**：
```
执行步骤 → 检查结果 → 触发反思 → 问LLM → LLM给建议 → 调整计划 → 继续执行
```

**LLM给的5种决策**：
1. **CONTINUE**：没问题，继续
2. **SKIP_NEXT**：下一步没必要，跳过
3. **ADD_STEPS**：缺了步骤，补上
4. **RETRY**：刚才没做好，重试
5. **FAIL**：彻底失败，放弃

**实际例子**：
假设要爬取网站数据：
- 第1步：分析网站结构 ✅ 成功
- 第2步：抓取数据 ❌ 失败（网站反爬）
- → 触发反思
- → LLM说："检测到反爬，建议添加代理步骤"
- → 系统自动插入"配置代理"步骤
- → 继续执行

就像有个教练在旁边看着你干活，发现不对就喊停并给建议。

## 4. **BFS双节点内容树** - 独特的文本处理架构

**核心设计**：层次化存储，支持长文本的结构化处理

**架构图**：
```
                              ┌─────────────────┐
                              │   Root Node     │  ← Level 0：根节点
                              │   (文档根节点)   │
                              └────────┬────────┘
                                       │
        ┌──────────────────────────────┼──────────────────────────────┐
        │                              │                              │
        ▼                              ▼                              ▼
┌─────────────┐              ┌─────────────┐              ┌─────────────┐
│   爬取      │              │   总结      │              │   搜索      │  ← Level 1：所有Skill功能节点
│  function   │              │  function   │              │  function   │
└──────┬──────┘              └─────────────┘              └─────────────┘
       │
       ├───┐
       │   │
       ▼   ▼
┌─────────┐ ┌─────────────────────┐
│ 概要1   │ │         内容1        │  ← Level 2：第1段双节点
│ summary │ │      content        │
└─────────┘ └──────────┬──────────┘
                       │
                       ├───┐
                       │   │
                       ▼   ▼
              ┌─────────┐ ┌─────────────────────┐
              │ 概要2   │ │         内容2        │  ← Level 3：第2段双节点
              │ summary │ │      content        │
              └─────────┘ └──────────┬──────────┘
                                     │
                                     ├───┐
                                     │   │
                                     ▼   ▼
                            ┌─────────┐ ┌─────────────────────┐
                            │ 概要3   │ │         内容3        │  ← Level 4：第3段双节点
                            │ summary │ │      content        │
                            └─────────┘ └─────────────────────┘
```

**设计优势**：
- 第1层（Level 0）：根节点
- 第2层（Level 1）：所有Skill功能节点（爬取、总结、搜索、分析、翻译等）
- 第3层及以下：双节点结构（左=概要，右=内容）
- 链式继承：内容节点的子节点是下一段的双节点

**用途**：
- 快速预览：只看summary节点
- 深度阅读：看content节点
- 智能检索：关键词匹配时可同时搜索摘要和原文

## 5. **灵活的结果聚合** - 把多个结果合成一个

**场景**：3个Agent分别给出了答案，怎么整合？

**4种策略**：

### ① 简单合并（SIMPLE_MERGE）
把所有结果打包在一起返回。
```
Agent1: "北京天气晴"
Agent2: "上海天气雨"
Agent3: "广州天气阴"
→ 返回：["北京天气晴", "上海天气雨", "广州天气阴"]
```

### ② 加权投票（WEIGHTED_VOTE）
按可信度加权。
```
Agent1（可信度90%）: "答案是A"
Agent2（可信度60%）: "答案是B"
Agent3（可信度80%）: "答案是A"
→ A得票 = 0.9 + 0.8 = 1.7
→ B得票 = 0.6
→ 最终答案：A
```

### ③ 层次聚合（HIERARCHICAL）
先分组再汇总。
```
按来源分组：
- Agent1组：3个结果，平均可信度85%
- Agent2组：2个结果，平均可信度70%
→ 总体报告：来自2个Agent的5个结果，整体质量77.5%
```

### ④ LLM总结（LLM_SUMMARIZE）⭐最智能
让LLM写一份完整报告。
```
Prompt: "以下是3个Agent的执行结果，请生成一份结构化报告..."
LLM输出:
## 执行摘要
本次任务共收集3个结果...

## 详细结果
1. 数据分析结果：...
2. 趋势预测：...

## 关键发现
- 发现1：...
- 发现2：...

## 建议
...
```

**怎么选策略**：
- 简单任务 → 简单合并
- 有冲突的答案 → 加权投票
- 多来源数据 → 层次聚合
- 需要专业报告 → LLM总结

## 6. **深度意图理解** - 读懂用户真正想要什么

**问题**：用户说"帮我查一下北京的天气"，系统需要理解：
- 要干什么？→ 查询
- 查什么？→ 天气
- 哪里？→ 北京
- 有什么要求？→ 尽快

**三层解析**：

### 第1层：意图分类
识别用户想干嘛：
- **QUERY**：查询信息（"什么是..."、"怎么..."）
- **ACTION**：执行动作（"运行"、"启动"）
- **CREATION**：创建内容（"写一篇..."、"生成..."）
- **ANALYSIS**：分析（"分析一下..."）
- ...等8种类型

**怎么判断**：
- 规则匹配：看到"查询"→QUERY，看到"分析"→ANALYSIS
- LLM辅助：复杂句子让LLM判断

### 第2层：实体提取
从句子中抽关键信息：
```
"帮我查一下2024年北京的天气"
→ 日期实体：2024年
→ 地点实体：北京
→ 主题实体：天气
```

支持提取：数字、日期、时间、邮箱、URL、文件路径、金额等。

### 第3层：约束识别
识别用户的要求：
```
"尽快给我一份高质量的JSON格式报告"
→ 时间约束：尽快（urgent）
→ 质量要求：高质量（high_quality）
→ 格式要求：JSON（structured_format）
```

### 最后输出
```python
ParsedIntent {
    intent_type: QUERY,           # 意图类型
    confidence: 0.85,             # 置信度85%
    primary_goal: "查北京天气",    # 主要目标
    entities: [                   # 提取的实体
        {name: "北京", type: "location"},
        {name: "天气", type: "topic"}
    ],
    constraints: {                # 约束条件
        time: "urgent",
        format: ["json"]
    },
    complexity: MEDIUM,           # 复杂度评估
    keywords: ["查", "北京", "天气"]
}
```

**有什么用**：
有了这个结构化信息，后续的任务分解、Agent选择、执行策略都能更精准。

## 7. **安全沙盒执行器** - 代码安全执行

**核心功能**：安全执行Python代码，防止恶意操作

**安全特性**：
- ❌ 禁止导入危险模块（os、sys、subprocess）
- ❌ 禁止eval/exec等危险函数
- ⏰ 超时保护（默认10秒）
- 💾 内存限制（默认256MB）

**执行流程**：
```
用户代码 → 代码校验 → 超时控制 → 执行 → 返回结果
              ↓
         拦截危险操作
```

**实际应用**：
```python
# 用户请求：计算数学表达式
result = sandbox.execute_python("1 + 2 * 3")
# 返回：{"success": True, "result": 7}

# 恶意代码被拦截
result = sandbox.execute_python("import os; os.system('rm -rf /')")
# 返回：{"success": False, "error": "危险操作被拦截"}
```

## 10. **完善的持久化** - Redis当仓库

**为什么需要持久化**：
- 系统重启后任务不能丢
- 多个服务实例要共享状态
- 分布式锁防止重复执行

**Redis里存什么**：

### ① 任务状态
```
Key: task:task_123
Value: {
    "status": "running",
    "goal": "爬取数据",
    "progress": 0.6,
    "created_at": 1234567890
}
```

### ② Agent状态
```
Key: agent:agent_456
Value: {
    "state": "busy",
    "current_task": "task_123",
    "load": 0.7
}
```

### ③ 上下文缓存（带过期时间）
```
Key: context:task_123
TTL: 3600秒（1小时后自动删除）
Value: {...对话历史...}
```

### ④ 分布式锁（防并发）
```
Key: lock:task_123
Value: uuid_xxx
TTL: 60秒

获取锁：SET lock:task_123 uuid_xxx NX EX 60
释放锁：用Lua脚本原子性删除
```

### ⑤ 消息队列（发布订阅）
```
发布：PUBLISH channel:tasks "新任务来了"
订阅：SUBSCRIBE channel:tasks → 收到通知
```

**好处**：
- 速度快（内存操作）
- 支持过期自动清理
- 原子操作保证一致性
- 天然支持分布式

## 12. **复杂协作模式** - Agent之间的"社交"

### ① 递归任务分解
**场景**：任务太复杂，一层分解不够。

```
原始任务："分析某公司财务状况"
↓ 第1层分解
- 收集财务数据
- 分析盈利能力
- 分析偿债能力
↓ 第2层分解（"收集财务数据"继续拆）
- 爬取年报
- 爬取季报
- 爬取新闻
↓ 第3层分解（"爬取年报"继续拆）
- 访问官网
- 下载PDF
- 解析表格
```

最多拆3层，避免无限递归。

### ② 动态团队组建
**场景**：不同任务需要不同技能的Agent组合。

```
任务："爬取并分析电商数据"
↓ 识别需求
需要：web_scraping + data_analysis + reporting
↓ 查找可用Agent
- Agent1（爬虫专家）：可用性90%，负载30%
- Agent2（数据分析师）：可用性85%，负载50%
- Agent3（报告生成器）：可用性95%，负载20%
↓ 组建团队
Team {
    members: [Agent1, Agent2, Agent3],
    role分配: {
        Agent1: scraper,
        Agent2: analyst,
        Agent3: reporter
    }
}
```

### ③ 任务拍卖机制
**场景**：多个Agent都能干，谁最合适？

**流程**：
```
1. 发布任务："爬取淘宝数据"
2. Agent投标：
   - Agent1出价：5元，预计10分钟，自信度90%
   - Agent2出价：8元，预计5分钟，自信度95%
   - Agent3出价：3元，预计20分钟，自信度70%
3. 评标（综合评分）：
   评分 = 自信度 × (1 - 出价/10) × (1 - 时间/30)
   - Agent1: 0.9 × 0.5 × 0.67 = 0.30
   - Agent2: 0.95 × 0.2 × 0.83 = 0.16
   - Agent3: 0.7 × 0.7 × 0.33 = 0.16
4. 中标：Agent1（性价比最高）
```

### ④ 共识机制
**场景**：多个Agent意见不一致，听谁的？

```
问题："这个数据是否异常？"
↓ 投票
Agent1: "是"（自信度90%）
Agent2: "是"（自信度85%）
Agent3: "否"（自信度60%）
↓ 统计
"是"得票率 = 2/3 = 67%
↓ 判断
阈值设为60% → 达成共识："是异常"
```

如果达不到阈值（比如3人各说各的），则判定"未达成共识"，需要人工介入或重新分析。

### ⑤ 知识共享
**场景**：Agent学到的经验，其他人也能用。

```
Agent1完成任务后：
→ 分享知识："爬取淘宝需要用Selenium，不能用requests"
→ 存入知识库：{
    knowledge_id: "kb_001",
    content: "淘宝反爬策略...",
    access_count: 0
}

Agent2遇到类似问题：
→ 查询知识库："淘宝 爬取"
→ 找到kb_001
→ 直接使用经验，少走弯路

热门知识排行：
1. kb_001（被访问50次）
2. kb_002（被访问30次）
...
```
